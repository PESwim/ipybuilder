SIMPLE - example¶

makehello.py –> examples/cmd:
  • …/makehello.exe
  • …/makehello.dll
  • …/StdLib.dll
Uses:
  • makehello.py
  • makehello_assembly.json
  • listreldll: (relative references running from builder/ directory):
    • examples/make.py
    • ipyver.py

Assembly file: - all false = standalone, embed, libembed

command-line: (from ../builder) - “” is for default_config file - True sets makeExe=True, make exe

python ipybuild.py "" examples/makehello.py examples/make_assembly.json True examples/cmd listdll=examples/listreldll.txt

import sys

# that works.

# *OR* works anywhere you have path to StdLib.dll

SIMPLE example - alternate relative¶

makehello.py –> examples/cmds:
  • …/makehello.exx
  • …/makehello.dll
  • …/StdLib.dll
Uses:
  • makehello.py
  • makehello_assembly.json
  • listdll: (relative references):
    • make.py
    • ..ipyver.py

Assembly file: - all false = standalone, embed, libembed

command-line: (from ../builder/examples)

python ../builder/ipybuild.py "" makehello.py make_assembly.json True cmds listdll=listdll.txt

NOTE:“cmds” in the above is the directory (next example will use default release by omitting. If instead of “cmds” we use “/cmds” the output directory will now be at root c:/cmds.

EMBED USER LIBS example¶

makehello_embed.py –> examples/release:
  • …/makehello_embed.exe
  • …/StdLib.dll
Uses:
  • makehello_embed.py
  • makehello_embed_assembly.json
  • listdll: (relative references):
    • make.py
    • ..ipyver.py

Assembly file: - standalone = false, embed= true, libembed = false

command-line: (from ../builder/examples) no arg for outdir so defaults to run /release

python ../ipybuild.py "" makehello_embed.py make_embed_assembly.json True listdll=listdll.txt

NOTE:“changed make_assembly to make_embed_assembly and omitted “cmd” in the above from previous.

THIS IS THE EMBEDED EXAMPLE- The .exe file has two embedded resources:

  • a blob IPDLL file which handles the ipy and sys overhead to start
  • a blob DLL.<project>DLL resources holding all the user libs

the user libs in this example are the make.py, ipyver.py –> to dll files this internal resource is auto-name makehello_embedDLL requires the main.py –> main.exe file to add the <project>DLL to the clr i.e. clr.AddReference("makehello_embedDLL")

Now we can refer directly to the methods in our Libs

ALTERNATIVELY - python ../ipybuild.py "" makehello_embed.py make_embed_assembly.json True listexe=listexe.txt

using the listexe key will cause our external .py modules to be compiled without the need for AddRef; however
our libs must be in the .py format and not .dlls.
NOTE: MUST change listdll.txt to listexe.txt - cant mixe the tags ‘exe’, ‘dll’, ‘zip’
Could also use exe=[ipyver.py,make.py] OR listexe=[ipyver.py,make.py] OR exe=listexe.txt

LIBEMBED USER LIBS example¶

makehello.py –> examples/release:
  • …/makehello_embed.exe
  • …/StdLib.dll

Assembly file: - standalone = false, embed= true, libembed = true

Changed to libembed = true in make_assembly.json

command-line: (from ../builder/examples) no arg for outdir so defaults to run /release

python ../ipybuild.py "" makehello.py make_assembly.json True listexe=listexe.txt

Note: Can Delete StdLib from /release directory - still have to clr.AddRef(“StdLib”) in main,
but now the StdLib is internal.

devop & debug: explore¶

Uses ipyver to track the loaded Assembly and module sources. This was useful to see how compiled versions were loaded/imported and show how to use these methods. This code all came from a one-line bug fix in the original IronPython Lib/ code, which kept the compiled standalones from running in an independent system. As I could not get IronPython to debug trace each line had to be checked manually in the entire Lib/ code.
